## Reflection

The DriverPass project centered around designing a system for a small driving school that wanted to improve how students prepare for their driving tests. The client needed a system that could support online practice exams, scheduling for on‑the‑road lessons, and secure access for different staff roles. My job was to take the information from the interview and turn it into clear requirements in Project One, then build out the technical design and UML diagrams in Project Two.

One thing I feel I did well was organizing the client’s needs into a structured set of functional and nonfunctional requirements. The interview had a lot of scattered information, but I was able to break it down into categories like scheduling, account management, security, and DMV updates. That made it easier to design the system later because everything connected back to what the client actually needed. I also think I did a solid job describing the technical setup in the system design document, especially the cloud‑based requirements and security expectations.

If I were to revise anything, I would probably improve the detail in my diagrams—especially the sequence diagram. I think adding more steps and interactions would make the workflow clearer for someone who might build the system later. I would also refine the class diagram to show more relationships and attributes so the structure of the system is easier to understand at a glance.

To interpret the user’s needs, I focused on the problems the client described, like low pass rates, scheduling issues, and the need for secure access. I translated those problems into system behaviors, such as tracking lesson assignments, locking accounts after failed logins, and updating DMV content. Considering user needs is important because a system can look good on paper but still fail if it doesn’t match how people actually work. Keeping the user’s goals at the center helps avoid that.

My approach to designing software is to start with requirements, then move into diagrams and structure. I like to break the system into smaller parts and understand how each component communicates. In the future, I plan to keep using techniques like use case modeling, activity diagrams, and iterative refinement because they help me stay organized and design systems that make sense.
